How to Use camelCase with FSCSS

6 min read · #camelCase #FSCSS #CSS #workflow

Let's face it: background-color feels nothing like the JavaScript you write every day. What if you could style elements using backgroundColor instead?

.card {
  backgroundColor: #1e293b;
  borderRadius: 16px;
  boxShadow: 0 4px 12px rgba(0,0,0,0.1);
}

With FSCSS's camelCase extension, this isn't just a dream — it's your reality. No build tools, no complex configs. Just import and write.

The Concept

From Hyphens to CamelCase

The camelCase extension is a lightweight .fscss module that maps JavaScript-style property names to their standard CSS equivalents. Each str() declaration creates a direct translation layer between your syntax and the browser.

/* Inside camelCase.fscss */
str(backgroundColor, "background-color")
str(borderRadius, "border-radius")
str(fontWeight, "font-weight")
str(marginTop, "margin-top")
str(paddingLeft, "padding-left")
str(boxShadow, "box-shadow")

Each str() pair is a promise: write the left token, get the right CSS property at compile time.

Setup

Importing the camelCase Extension

FSCSS automatically resolves camelCase from its CDN. You have two import styles depending on your FSCSS version.

Modern Import (version 1.1.16 up)

@import((*) from camelCase)

The wildcard (*) imports all property mappings. You can also import selectively:

@import((camelCase) from camelCase)

exec Method (legacy)

@import(exec(_init camelCase))

The _init prefix tells FSCSS to load the extension from the CDN without specifying the .fscss file extension.

Complete setup example:

<script src="https://cdn.jsdelivr.net/npm/fscss@1.1.22/exec.min.js" defer></script>
<style>
  @import((*) from camelCase)
  
  body {
    backgroundColor: #f8fafc;
    fontFamily: 'DM Sans', sans-serif;
  }
</style>

In Practice

Real-World Usage Examples

Once imported, camelCase property names work anywhere a standard CSS property would. Here are common patterns.

Button Component

.btn-primary {
  backgroundColor: #3b82f6;
  color: white;
  borderRadius: 8px;
  paddingTop: 10px;
  paddingBottom: 10px;
  paddingLeft: 20px;
  paddingRight: 20px;
  fontWeight: 500;
  transition: all 0.2s ease;
  
  &:hover {
    backgroundColor: #2563eb;
    transform: translateY(-2px);
  }
}

Card Layout

.card {
  backgroundColor: white;
  borderRadius: 24px;
  boxShadow: 0 10px 25px -5px rgba(0,0,0,0.1);
  padding: 24px;
  marginBottom: 20px;

.card-title {
  fontSize: 1.5rem;

  marginBottom: 12px;
  lineHeight: 1.2;

.card-description {
  fontSize: 1rem;

Responsive Design

Extending

Adding Your Own camelCase Properties

The camelCase extension covers standard properties, but you can extend it with your own str() mappings for custom needs or proprietary CSS features.

/* Add to your own extension or after import */
str(backdropBlur, "backdrop-filter: blur(8px);")
str(textGradient, "background: linear-gradient(135deg, #667eea, #764ba2); -webkit-background-clip: text; background-clip: text; color: transparent;")

.hero {
  backdropBlur
  textGradient
}

Notice how str() can map a single token to multiple declarations — perfect for creating design system shorthands.

The Why

Benefits of camelCase CSS

  • Consistency with JavaScript: React, Vue, and vanilla JS all use camelCase for inline styles — now your stylesheets match.
  • Less cognitive friction: No more switching between hyphens and camelCase when moving between JS and CSS.
  • IDE autocomplete: Many editors provide better autocomplete for camelCase tokens after the extension loads.
  • Team alignment: Frontend teams already familiar with JS conventions adopt CSS faster.
  • No build tools: FSCSS works at runtime or compile-time — zero config overhead.

camelCase CSS isn't just about aesthetics — it's about reducing the mental tax of switching between languages.

Troubleshooting

Common Questions

Q: Does camelCase work with CSS variables?
Yes. var(--my-color) works exactly as expected. The extension only transforms property names, not values.

Q: Can I mix camelCase and standard CSS?
Absolutely. The extension adds new valid tokens without removing existing CSS syntax.

Q: What about vendor prefixes?
Standard prefixes work: WebkitBackdropFilter maps to -webkit-backdrop-filter.

Q: Is there a performance cost?
The transformation happens once at compile/load time. Runtime performance is identical to standard CSS.
