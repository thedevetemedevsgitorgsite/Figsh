How to Use snake_case with FSCSS

6 min read · #snake_case #FSCSS #CSS #python

If you're a Python developer or simply love the clean, readable style of snake_case, CSS's hyphenated properties can feel out of place. What if you could write border_radius and background_color instead?

.dashboard-card {
  background_color: #1e293b;
  border_radius: 16px;
  box_shadow: 0 4px 12px rgba(0,0,0,0.1);
  padding_top: 20px;
  padding_bottom: 20px;
}

With FSCSS's snake_case extension, you can write CSS using Pythonic naming conventions. Import once, and your stylesheets will feel right at home alongside your backend code.

The Concept

From Hyphens to Underscores

The snake_case extension is a lightweight .fscss module that maps Python-style property names to their standard CSS equivalents. Each str() declaration creates a direct translation layer between your syntax and the browser.

/* Inside snake_case.fscss */
str(background_color, "background-color")
str(border_radius, "border-radius")
str(font_weight, "font-weight")
str(margin_top, "margin-top")
str(padding_left, "padding-left")
str(box_shadow, "box-shadow")
str(text_align, "text-align")
str(line_height, "line-height")

Each str() pair transforms snake_case tokens into standard CSS properties at compile time — zero runtime overhead.

Setup

Importing the snake_case Extension

FSCSS automatically resolves snake_case from its CDN. You have two import styles depending on your FSCSS version.

Modern Import (v1.1.16+)

@import((*) from snake_case)

The wildcard (*) imports all property mappings. You can also import selectively:

@import((snake_case) from snake_case)

exec Method (legacy)

@import(exec(_init snake_case))

The _init prefix tells FSCSS to load the extension from the CDN without specifying the .fscss file extension.

Complete setup example:

<script src="https://cdn.jsdelivr.net/npm/fscss@1.1.22/exec.min.js" async></script>
<style>
  @import((*) from snake_case)
  
  body {
    background_color: #f8fafc;
    font_family: 'DM Sans', sans-serif;
    line_height: 1.6;
  }
</style>

Extending

Adding Your Own snake_case Properties

The snake_case extension covers standard properties, but you can extend it with your own str() mappings for custom needs or proprietary CSS features.

/* Add to your own extension or after import */
str(glass_effect, "backdrop_filter: blur(12px); background: rgba(255,255,255,0.2);")
str(gradient_text, "background: linear-gradient(135deg, #667eea, #764ba2); -webkit-background-clip: text; background-clip: text; color: transparent;")

.hero_section {
  glass_effect
  gradient_text
}

Notice how str() can map a single snake_case token to multiple declarations — perfect for creating design system shorthands that match your naming conventions.

The Why

Benefits of snake_case CSS

  • Python ecosystem alignment: If you're building Django, Flask, or FastAPI apps, your CSS now matches your backend naming conventions.
  • Enhanced readability: Underscores can make long property names easier to parse at a glance.
  • Consistent with databases: Many SQL and data workflows use snake_case — now your frontend matches.
  • Team familiarity: Teams coming from Python or Ruby backgrounds adopt CSS faster with familiar syntax.
  • No build tools: FSCSS works at runtime or compile-time — zero config overhead.

snake_case CSS bridges the gap between frontend and backend naming conventions, creating a more unified codebase experience.

Comparison

Choosing Your Naming Style

FSCSS offers both camelCase and snake_case extensions. Which one should you choose? It depends on your team's background and project context.

camelCase

backgroundColor borderRadius marginTop

Best for: React/Vue/Next.js projects, JavaScript-heavy teams, inline styles consistency.

snake_case

background_color border_radius margin_top

Best for: Python/Django backends, data-heavy applications, teams preferring readability.

You can even use both in the same project — FSCSS allows mixing extensions freely.

Troubleshooting

Common Questions

Q: Does snake_case work with CSS custom properties (variables)?
Yes. var(--primary_color) works exactly as expected. The extension only transforms property names, not values or variable references.

Q: Can I mix snake_case and standard CSS?
Absolutely. The extension adds new valid tokens without removing existing CSS syntax. You can use background-color and background_color side by side.

Q: What about vendor prefixes?
Standard prefixes work: webkit_backdrop_filter maps to -webkit-backdrop-filter.

Q: Is there a performance cost?
The transformation happens once at compile/load time. Runtime performance is identical to standard CSS.

Q: Can I use snake_case in media queries and keyframes?
Yes. All standard CSS at-rules support snake_case property names after import.
