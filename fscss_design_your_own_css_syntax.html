FSCSS: What If You Could Design Your Own CSS Syntax?

5 min read · #CSS #FSCSS #DevTools #opensource

A few days ago, I wrote this inside a stylesheet:

.card { my-card-x }

No class utilities. No framework. No JavaScript.

And it worked — not because CSS supports it, but because I defined what my-card-x means. That's the idea behind FSCSS.

The Shift

From Writing CSS to Designing CSS

Traditional CSS gives you a fixed vocabulary. You memorise its properties and work within them. FSCSS flips the premise entirely: instead of adapting to CSS, you define what CSS means inside your project.

Traditional

background-color: red; border-radius: 8px; margin-top: 20px;

FSCSS

backgroundColor: red; bdRadius: 8px; my-card-x;

You can redefine how CSS itself is written — and that changes everything about how you architect styles.

Core Primitive

str() — The Foundation

At its simplest, str() maps one syntax token to another. It's a translation layer sitting between your stylesheet and the browser.

str(backgroundColor, "background-color")

After that single declaration, this is valid:

body { backgroundColor: red; }

But str() isn't limited to single-property aliases. It can expand a single token into an entire block of styles — turning a bare identifier into a full styling primitive:

str(my-card-x, " background: red; color: white; padding: 20px; border-radius: 20px; ")

This is no longer shorthand — it's a custom styling primitive you invented.

A Cleaner Abstraction

Parameterized Primitives with @define

Once you want to pass data into your custom syntax, @define is the right tool. It introduces parameters, structure, and reuse — without leaving CSS-like syntax.

@define card(bg, color) { background: @use(bg); color: @use(color); padding: 20px; border-radius: 20px; }

Then use it anywhere:

.card { @card(#ff9955, white); }

Think of it like CSS functions you wrote yourself. No build plugins. No PostCSS config. Just FSCSS.

Extensions & Modules

The Import System

FSCSS extensions are plain .fscss files. There are two import styles, each suited to different situations:

StyleSyntaxBest for
exec (standard)@import(exec("location"))Debugging, custom paths, external files
Named (wildcard)@import((*) from "location")Importing everything from a module
Named (selective)@import((blocks) from "location")Importing specific blocks only

Built-in CDN Resolution

FSCSS automatically resolves imports against its CDN. So this:

@import(exec(_init camelCase))

…loads camelCase.fscss straight from the CDN. The small but important rule: _init is used with exec, but dropped when using from.

In Practice

A Real Extension: camelCase.fscss

A typical extension file is just a list of str() mappings. The camelCase module maps JavaScript-style property names to their CSS equivalents:

str(backgroundColor, "background-color") str(borderTopRightRadius, "border-top-right-radius") str(fontWeight, "font-weight")

After importing it, camelCase property names become valid anywhere in your stylesheets:

body { backgroundColor: #FF9955; fontWeight: 600; }

The Bigger Picture

What FSCSS Enables

  • Create your own property naming conventions — camelCase, snake_case, or totally custom tokens
  • Build shorthand systems precisely tailored to your team's workflow
  • Define reusable style primitives that encapsulate entire design decisions
  • Share extensions as plain .fscss files — no package overhead
  • Extend CSS expressiveness without touching JavaScript or a build config
